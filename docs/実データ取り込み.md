# 実データの取り込み（嘉島町 介護予防健診データ）

既存の Google スプレッドシート（★介護予防健診_Crutoデータベース）を、Cruto motion に取り込んで
**ログイン内でのみ表示・編集**するための流れ。個人情報を含むため、リポジトリや公開デモには一切載せない。

```
xlsx（Googleスプレッドシート書き出し）
  → ① ETL 正規化   functions/scripts/etl-kashima.py  → normalized.json（PII・コミット禁止）
  → ② Firestore投入 functions/scripts/seed-firestore.mjs → users / measurements
  → ③ アプリが認証後に Firestore から読み込み（src/lib/realdata.js）→ 台帳・個人票・集計に表示
```

## 取り込み元と正本（46タブから）

| 用途 | 正本タブ | 備考 |
|---|---|---|
| 名簿 | `データベース`(2025) | 氏名/かな/性別/生年月日/身長体重 |
| 2024 測定 | `参照_利用者データ` | IDキー・ファイルメーカー形式 |
| 2025 測定 | `介護予防検診_身体機能` | フォーム回答。5m最大・左右別はここが唯一の正本 |
| InBody SMI | `2024データベース` / `2025inbody` | |
| 行政区 | `マスタ` | 測定日→地区（下六嘉・犬渕 等） |

`データベース` の測定列は空（名簿のみ）。`県提出用◯◯`（約30タブ）は提出スナップショットで取り込み対象外。

## 手順

```bash
# ① 正規化（xlsx と同じ場所で）
pip install openpyxl
python3 functions/scripts/etl-kashima.py ./cruto-db.xlsx     # → normalized.json を生成

# ② Firestore へ投入
#   ローカル検証（エミュレータ）
FIRESTORE_EMULATOR_HOST=127.0.0.1:8080 GOOGLE_CLOUD_PROJECT=demo-cruto \
  node functions/scripts/seed-firestore.mjs ./normalized.json
#   本番（要 ADC: gcloud auth application-default login）
GOOGLE_CLOUD_PROJECT=YOUR_PROJECT node functions/scripts/seed-firestore.mjs ./normalized.json
```

③ は自動。`VITE_FIREBASE_CONFIG` を設定したビルドで、職員ログイン後に `loadRealData()` が
Firestore から読み込み、エンジンの利用者配列を実データに差し替える。未設定（公開デモ）は従来のシードのまま。

## Firestore データモデル

| コレクション | 主なフィールド |
|---|---|
| `users/{id}` | `name` `kana` `sex` `birthDate` `muni` `muniName` `ward` `careLevel` `phone` `flags` |
| `measurements/{id}_{年}` | `userId` `year` `date` `values{walk5,walk5max,balR,balL,gripR,gripL,tug,height,weight,bmi}` `inbodySmi` `review` |

`review:true` は年度間で身長が大きく食い違う「要確認」測定（旧年度データの取り違え疑い）。

## ローカル検証（エミュレータで表示まで確認）

```bash
cd functions && npx firebase emulators:start --only firestore,auth --project demo-cruto   # 別ターミナル
# テストアカウント作成 + データ投入（上記 ②）
# フロントをエミュレータに向けて起動
VITE_FIREBASE_CONFIG='{"apiKey":"fake-api-key","projectId":"demo-cruto","authDomain":"demo-cruto.firebaseapp.com","appId":"1:1:web:demo"}' \
VITE_AUTH_EMULATOR_URL='http://127.0.0.1:9099' VITE_FIRESTORE_EMULATOR='127.0.0.1:8080' npm run dev
```

## 公開デモと本番（2サイト構成）

| | URL | 中身 | ログイン |
|---|---|---|---|
| 公開デモ | konohito.github.io/c-crutomotin | シード（架空） | 不要 |
| 本番 | cruto-motion.web.app（Firebase Hosting） | 実データ（Firestore） | 必須 |

- 公開デモ = 既定の `npm run build`（`VITE_FIREBASE_CONFIG` 無し・base=`/c-crutomotin/`）。無変更
- 本番 = `npm run build:hosting`（`--mode hosting` → `.env.hosting` の `VITE_FIREBASE_CONFIG` を読み、base=`/`）
- 本番デプロイ: `.github/workflows/deploy-hosting.yml`（Secrets/Variables 設定後はマージで自動）。
  手動なら `npm run build:hosting && npx firebase deploy --only hosting --project cruto-motion`

## 職員アカウントの承認（セキュリティ必須）

Firestore ルールは `staff/{uid}` を持つ職員だけに実データを許可する（apiKey は公開情報のため
「ログイン済み」だけでは不十分）。手順:

1. Firebase Authentication で職員アカウントを作成（メール/パスワード）
2. `GOOGLE_CLOUD_PROJECT=cruto-motion node functions/scripts/grant-staff.mjs <email> "氏名"`
   （承認解除は `<email> --revoke`）

未承認のアカウントはログインできても「閲覧権限がありません」画面になり、データは一切出ない。

## 注意

- `normalized.json` / `*.xlsx` / `contradictions.txt` は個人情報。`.gitignore` 済み。コミットしない
- `.env.hosting` の Firebase 構成は公開情報（クライアント埋め込み前提）。保護はルール＋承認で担保
- カレンダー・測定会イベントは現状シードのまま（実データ側の移行は別スコープ）
- 取り込み後の値はアプリ上で編集・修正できる（`要確認` の値もここで直す）
